# ADR-0019: Test Runner Migration from Karma to Jest

**Status:** Accepted
**Date:** 2026-06-14

## Context

Karma was the default test runner generated by `ng new` when the project was scaffolded.
No conscious architectural decision was made to choose it — it was the Angular CLI default
at the time. Karma spawns a real browser (ChromeHeadless) for every test run, which carries
two consequences:

1. **CI cost:** Each `ng test` invocation starts Chrome, loads Zone.js, and bootstraps Angular
   even for pure domain unit tests with no DOM interaction.
2. **Framework trajectory:** The Angular team deprecated Karma starting with Angular 16 and
   recommends Jest or Web Test Runner as replacements for all future projects.

Alternatives considered at migration time:

- **Stay on Karma** — becomes unsupported as Angular removes the karma builder in a future
  major version; no improvement to CI speed or developer loop
- **Web Test Runner (WTR)** — browser-based like Karma, avoids jsdom gaps; no official Angular
  builder in v18; fewer community resources for Angular-specific patterns
- **Jest via `@angular-devkit/build-angular:jest`** — experimental builder shipped with Angular 18;
  too restrictive: no `setupFilesAfterEnv`, no coverage flags, no arbitrary Jest CLI args
- **Jest standalone (`npx jest`)** — runs in Node.js via jsdom; full Jest API surface; `jest-preset-angular`
  handles Angular 18 compilation and `TestBed` initialization; actively recommended by the Angular team
- **Vitest** — no Angular builder; requires a Vite-based project setup; not compatible with
  Angular CLI's webpack builder in Angular 18

## Decision

Migrate the test runner from **Karma + Jasmine** to **standalone Jest** (`npx jest`) using
`jest-preset-angular` as the compiler and test environment bridge.

The Angular builder (`@angular-devkit/build-angular:jest`) was evaluated and rejected: its
JSON schema does not expose `setupFilesAfterEnv`, preventing a shared global setup file, and
it does not support coverage flags or arbitrary Jest CLI arguments needed for CI integration.

### Configuration changes

| File | Change |
|---|---|
| `jest.config.js` | New file — `preset: 'jest-preset-angular'`, `setupFilesAfterEnv`, coverage config |
| `setup-jest.ts` | New file — `setupZoneTestEnv()` + `crypto.randomUUID` polyfill (see below) |
| `angular.json` | `test:` builder removed entirely — standalone Jest replaces `ng test` |
| `package.json` | `"test": "npx jest"`. Removed: `karma`, `karma-chrome-launcher`, `karma-coverage`, `karma-jasmine`, `karma-jasmine-html-reporter`, `@types/jasmine`. Added: `jest`, `jest-environment-jsdom`, `jest-preset-angular`, `@types/jest` |
| `tsconfig.spec.json` | `types: ["jasmine"]` → `types: ["jest"]` |
| `.github/workflows/ci-cd.yml` | `ng test` → `npx jest --coverage`; removed Karma-specific flags (`--browsers=ChromeHeadless`, `--no-progress`, `--watch=false`, `--code-coverage`) |

### Jasmine → Jest API mapping (applied across 14 spec files)

| Jasmine | Jest |
|---|---|
| `jasmine.SpyObj<T>` | `jest.Mocked<T>` |
| `jasmine.createSpyObj('name', [...methods])` | Manual object with `jest.fn()` per method |
| `.and.returnValue(x)` | `.mockReturnValue(x)` |
| `jasmine.createSpy('name')` | `jest.fn()` |
| `jasmine.any(T)` | `expect.any(T)` |
| `jasmine.objectContaining({...})` | `expect.objectContaining({...})` |
| `.calls.mostRecent().args[0]` | `.mock.lastCall[0]` |
| `toBeTrue()` / `toBeFalse()` | `toBe(true)` / `toBe(false)` |
| `toHaveBeenCalledOnceWith(x)` | `toHaveBeenCalledTimes(1)` + `toHaveBeenCalledWith(x)` |
| `toThrowMatching(e => e instanceof T && e.code === X)` | `try/catch` + `toBeInstanceOf(T)` + property assertion on caught error |

### Technical blockers resolved

| # | Symptom | Root cause | Fix |
|---|---|---|---|
| 1 | `TS2352: Conversion of type '{}' to type 'Mocked<T>' may be a mistake` | Partial mock objects are missing private fields (`repository`, `eventBus`) that TypeScript includes in the structural shape | `as unknown as jest.Mocked<T>` double-cast on all mock objects |
| 2 | `Property 'toThrowMatching' does not exist` | `toThrowMatching` is Jasmine-only; Jest has no equivalent matcher | `try/catch` block with `toBeInstanceOf` + property assertion on the caught error |
| 3 | `crypto.randomUUID is not a function` | jsdom does not implement `crypto.randomUUID`; production code calls it for aggregate ID generation | Global polyfill in `setup-jest.ts` via `node:crypto` (works in standalone Jest; blocked by Angular bundler in builder mode) |
| 4 | `The package "node:crypto" wasn't found on the file system` | Angular's builder treats setup files as browser targets; `import from 'node:crypto'` is rejected | Switched to standalone Jest — setup files run in Node.js, so `node:` protocol imports work |
| 5 | `Schema validation failed: must NOT have additional properties (setupFilesAfterEnv)` | `@angular-devkit/build-angular:jest` builder does not expose `setupFilesAfterEnv` in its JSON schema | Removed the Angular builder entirely; standalone Jest supports the full Jest config API |

### `setup-jest.ts`

```typescript
import { setupZoneTestEnv } from 'jest-preset-angular/setup-env/zone';
setupZoneTestEnv();

// jsdom does not implement crypto.randomUUID — required by services that generate aggregate IDs
import { randomUUID } from 'node:crypto';
Object.defineProperty(globalThis, 'crypto', {
  value: { randomUUID },
  configurable: true,
  writable: true,
});
```

`setupZoneTestEnv()` replaces the deprecated `import 'jest-preset-angular/setup-jest'` API
(removed in jest-preset-angular v14) and calls `TestBed.initTestEnvironment()` with Zone.js.

## Consequences

**Positive:**
- No browser startup — `npx jest` completes in ~9 s vs. Karma's ~30 s on the same 148 tests
- Lower CI memory — no Chrome process; fits comfortably within GitHub Actions' 7 GB runner
- Faster local watch mode — Jest restarts in milliseconds, no browser cold-start
- `--testPathPattern` and `--testNamePattern` allow targeting a single spec or test case
  without modifying source files
- `jest.Mocked<T>` carries more type information than `jasmine.SpyObj<T>` — TypeScript surfaces
  mock shape mismatches at compile time, not at runtime
- `crypto.randomUUID` polyfill is defined once in `setup-jest.ts`; no per-file `beforeAll` burden

**Negative:**
- jsdom does not emulate the full browser API — tests relying on canvas, WebGL, or complex
  CSS layouts would still need a real browser runner; no such tests exist in this project currently
- `ng test` no longer works — the test builder was removed from `angular.json`; the test command
  is now `npx jest` (or `npm test`)
- **ADR-0020 (Stryker) must be updated:** the Stryker `commandRunner.command` currently
  passes `--browsers=ChromeHeadless` (Karma-specific, now invalid) and `--no-code-coverage`
  (Karma/Istanbul flag; Jest equivalent is `--no-coverage`); both flags must be updated
  when the mutation testing workflow is migrated to the Jest runner
